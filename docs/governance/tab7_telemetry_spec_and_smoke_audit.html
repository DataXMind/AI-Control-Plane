ADD ARCHITECTURE.md — hash-chain contract (1 đoạn)
Thêm section "Telemetry Hash-Chain Contract": ## Telemetry hash-chain contract Every `TelemetryEvent` appended via `InMemoryTelemetryStore.append()` is sealed: - `event_hash` = SHA-256(previous_hash + canonical_json(event, exclude=[event_hash, previous_hash, id])) - `previous_hash` = previous event's `event_hash` (empty string for first event) - Chain is verifiable via `verify_event_chain(store.list_events())` Invariant: `TelemetryWriter.emit()` is the ONLY public way to append events. Direct `store._events.append()` in production code is forbidden — breaks chain integrity. Telemetry failure MUST NOT affect PolicyEngine decisions (fail-silent, never fail-closed override).
API contract decision
KHÔNG cần POST /telemetry/ingest — Milestone A
In-process store đủ cho PoC. TypeScript bridge không cần telemetry endpoint — chỉ cần policy evaluate. Thêm endpoint khi Milestone B có Redis store và TS bridge cần query audit log.
Justification đầy đủ
Lý do 1
TypeScript bridge không cần gọi telemetry
Theo ARCHITECTURE.md Invariant #6: api/ là cross-language bridge duy nhất. TypeScript PolicyClient gọi POST /policy/evaluate. Telemetry là in-process concern của Python layer — TS không cần đọc hay ghi event chain. Thêm endpoint = surface attack không cần thiết.
Lý do 2
In-memory store đủ cho PoC audit
Milestone A mục tiêu: verify chain integrity và MCP events được sealed. InMemoryTelemetryStore với verify_chain() đáp ứng đủ. Data không persist qua restart — expected behavior Milestone A.
Lý do 3
Endpoint Milestone B — khi có Redis + audit query
Khi Milestone B có RedisQuotaStore và persistent store, thêm GET /telemetry/events?project_id=&type= để operator query audit log. Schema: re-export TelemetryEvent từ api/schemas.py với class TelemetryEventResponse(TelemetryEvent): pass — không wrapper riêng.
Lý do 4
Fail-silent contract rõ ràng hơn
Không có HTTP endpoint → không có timeout risk trên telemetry path. Telemetry failure = structlog error + continue. PolicyDecision không bị ảnh hưởng. Nếu có endpoint → risk timeout ảnh hưởng latency của policy path.
Blind spot assessment (BS-02, BS-11, BS-13, BS-15)
BS-02
Model drift — TelemetryEvent fields thay đổi
compute_event_hash exclude event_hash, previous_hash, id. Nếu ai thêm field mới vào TelemetryEvent mà không update exclude list → hash thay đổi → chain break silently. Mitigation: TC-1 test determinism sẽ bắt — nhưng chỉ khi field mới có giá trị thay đổi giữa runs. Recommendation: thêm comment trong compute_event_hash liệt kê rõ excluded fields và lý do.
BS-11
Registry/telemetry isolation — store không shared với registry
InMemoryTelemetryStore và AgentRegistry phải là separate singletons trên AppState. Risk: nếu Cursor inject store sai instance → events không trong chain. Mitigation: AppState là single source — git_server nhận app_state.telemetry_store, không tạo new store. Test: TC-7 qua TelemetryWriter inject store từ AppState.
BS-13
Thread safety — InMemoryTelemetryStore concurrent appends
FastAPI async + uvicorn workers có thể concurrent append. Nếu InMemoryTelemetryStore dùng plain list → race condition trên _events.append() + _last_hash update. Risk: chain corruption khi high concurrent tool calls. Mitigation Milestone A: thêm threading.Lock vào append(). Test: không cần concurrent test cho Milestone A — document risk trong ARCHITECTURE.md.
BS-15
Regression — SMK-01..05 vẫn pass sau MCP patch
SMK-03 test policy allow qua GitMcpServer. Nếu MCP patch thay đổi constructor signature → create_app() tạo GitMcpServer sai → SMK-03 fail. Mitigation: backward compat constructor (store=None default), tạo local store nếu None. Test gate: pytest -m smoke phải pass sau patch trước khi commit.
