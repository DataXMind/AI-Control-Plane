Nguyên tắc vàng khi dùng Cursor cho dự án này: Cursor hoạt động tốt nhất khi nó biết toàn bộ kiến trúc trước, rồi mới nhận lệnh từng file nhỏ. Không bao giờ nói "build cả project". Mỗi prompt = 1 file hoặc 1 class.
Luồng làm việc 3 tầng

Tầng 1 — Context (làm 1 lần)

Tạo .cursorrules + ARCHITECTURE.md. Đây là "bộ nhớ dài hạn" cho mọi session Cursor về sau.

Tầng 2 — Session prompt (mỗi session)

Mở session mới → paste "Session Context Prompt" ngắn → Cursor biết ngay đang làm gì, không bị lạc.

Tầng 3 — Task prompt (mỗi file)

Prompt 1 file/class cụ thể. Ngắn, có spec rõ, có test expectation. Không ambiguous.

Quy tắc session

Mỗi Cursor session = 1 module (vd: core/). Đừng mix session. Khi sang module mới → session mới + context prompt mới.

Thứ tự xây dựng (theo Milestone A)
Thứ tự thực hiện
Step 1 (trước khi mở Cursor) → Tạo repo + pyproject.toml + .cursorrules + ARCHITECTURE.md Step 2 — Session A: Config layer → config/loader.py (load + validate 3 yml files) Step 3 — Session B: Data contracts → core/models.py (union V1+V2 types — 11 classes) Step 4 — Session C: Registry + Policy + Quota → core/registry.py → core/policies.py → core/quota.py (thứ tự quan trọng: registry trước, policies sau) Step 5 — Session D: Telemetry → core/telemetry.py (TelemetryEmitter, EventStore, AuditWriter) Step 6 — Session E: MCP layer → mcp/git_server.py → mcp/server_factory.py → mcp/server_utils.py Step 7 — Session F: API bridge → api/schemas.py → api/server.py (5 endpoints) Step 8 — Session G: CLI → main.py → cli/assign.py → cli/status.py → cli/logs.py Step 9 — Session H: Tests → conftest.py → test_config_loader.py → test_models.py → ... Step 10 — apex/ stubs (import-clean, không logic) → apex/__init__.py + 5 stub files

# ai-control-plane .cursorrules # Đọc ARCHITECTURE.md để hiểu đầy đủ context trước khi generate code. ## Project identity - Python package: ai_control_plane (src/ layout) - CLI: agentctl (Typer) - API: FastAPI at api/server.py - Config: Pydantic v2 + PyYAML, path via ACP_CONFIG_DIR env var ## Code generation rules ALWAYS: - Dùng Pydantic v2 syntax (model_config = ConfigDict(...), không class Config) - Type hint đầy đủ tất cả function signatures - Dùng structlog thay vì logging.getLogger - Async/await cho tất cả I/O operations - Docstring 1 dòng cho mỗi public class và function NEVER: - Import OSS agent frameworks (CrewAI, LangChain, AutoGen) vào bất kỳ core/ file nào - Viết Git logic trong Python (chỉ có trong TypeScript cyanheads fork) - Hardcode Redis connection string (phải qua env var hoặc constructor injection) - Default-allow khi policy evaluation fail (luôn fail-closed) - Import trực tiếp từ core/policies.py trong cli/ (phải qua api/ HTTP) ## File ownership rules - core/models.py: source of truth cho types — import từ đây, không tạo duplicate - core/policies.py: pure logic, không I/O — test bằng cách inject mock QuotaStore - api/server.py: HTTP interface duy nhất cho TypeScript bridge - mcp/git_server.py: facade only — không business logic Git ## Test requirements Mỗi file trong core/ phải có test file tương ứng trong tests/. Test file phải dùng fixtures từ conftest.py. Không mock PolicyEngine trong integration tests — dùng real instance với test yml. ## Naming conventions - Classes: PascalCase - Functions/vars: snake_case - Constants: UPPER_SNAKE_CASE - Private: _leading_underscore - Config env vars: ACP_ prefix (ACP_CONFIG_DIR, ACP_API_HOST, ACP_LOG_LEVEL) ## Error handling pattern - Domain errors: custom exceptions kế thừa từ ControlPlaneError base class - HTTP errors: HTTPException với status_code + detail JSON - MCP errors: McpError(code, message, data) — không raise raw exceptions - Luôn log trước khi raise: logger.error("...", exc_info=True) ## Milestone guard Apex/ files ở Milestone A: chỉ được có __init__.py và stub files với NotImplementedError. Không viết logic thực trong apex/ cho đến khi Milestone C bắt đầu.
