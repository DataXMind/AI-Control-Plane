Karpathy 4 principles = Layer 0 của 6-layer architecture đề xuất. Repo này là behavioral constitution cho AI coding agent — 175k stars vì nó giải quyết đúng vấn đề LLM drift. Câu hỏi: ACP đã có bao nhiêu, và còn thiếu gì?
4 principles — ACP đã có bao nhiêu?
Think Before
ACP có: PACE (Plan → Act → Check → Evolve) trong DEVELOPMENT_PROTOCOL
75% có DEVELOPMENT_PROTOCOL.md ghi rõ "Plan phase" trước implement. Claude architect review → Cursor execute. Nhưng thiếu: explicit assumption surfacing. Nhiều lần Cursor implement với silent assumption (agent4 scope creep, monolithic PR). Gap: không có structured "state assumptions before coding" step trong .cursorrules.
Simplicity
ACP có: "Surgical changes" trong .cursorrules hiện tại
50% có .cursorrules ghi "Không thêm feature không được yêu cầu". Nhưng thực tế: monolithic PR #48 chứa MB-S1-1..5 + agent4 (scope creep), MC-1..11 gộp 1 PR. Karpathy: "if 200 lines could be 50, rewrite it" — ACP không có enforcement mechanism cho điều này. Gap: không có diff size limit, không có abstract complexity check.
Surgical
ACP có: Invariant #2 (models.py only), #4 (CLI HTTP only)
80% có 8 invariants là surgical constraint rõ nhất. Nhưng: "touch only what you must" không được enforce ở task level. Agent4 config merge vào PR #46 (doc-only PR) là classic orthogonal edit. Gap: không có file allowlist/denylist per task type trong .cursorrules.
Goal-Driven
ACP có: acceptance criteria trong mỗi sprint prompt, SMK gates
85% có Đây là điểm mạnh nhất của ACP. Mỗi Cursor prompt có verify gate rõ. SMK-01..08 là success criteria verifiable. Karpathy: "give it success criteria and watch it go" — ACP đã làm tốt. Chỉ cần formal: mọi task phải có "verify: [command]" trong prompt.
Insight quan trọng nhất từ Karpathy cho ACP
Vấn đề gốc rễ của ACP drift không phải là thiếu rules — mà là rules không được structured đúng layer. .cursorrules hiện tại là flat list ~30 rules. Khi Cursor không rõ priority, nó silent-picks interpretation. 6-layer architecture giải quyết điều này bằng cách tạo rõ: Layer 0 luôn thắng Layer 3, Layer 2 policy beats Layer 3 guardrails nếu conflict. Đây là thiếu sót kiến trúc lớn nhất của ACP hiện tại.

Workflow thay đổi — Claude ↔ Cursor interaction
Trước
Claude write prompt → Cursor execute → verify gate
Flat workflow. Cursor nhận task, không có risk triage. Kết quả: scope creep (agent4), monolithic PR, silent assumption.
Sau
Claude write prompt (với L2 risk header) → Cursor state assumptions → execute → verify gate
Thêm 2 bước: (1) Cursor phải state risk level + assumptions trước khi code, (2) Mỗi prompt có explicit file allowlist. Nếu task = HIGH risk → Claude review mid-task, không chỉ post-task.
L5 loop
Failure → LESSONS_LEARNED.md → update .cursorrules layer tương ứng
Hiện tại: failures ghi trong audit reports nhưng không có formal feedback loop vào .cursorrules. Sau: mỗi sprint end có "lessons triage" step — extract pattern → assign to layer → update rule.
Phân loại artifacts hiện có theo 6-layer fit
FITS L0
.cursorrules — nhưng cần restructure thành 6 sections
Hiện tại là flat list. Karpathy đề xuất: behavioral rules ở top, project rules ở giữa, forbidden list explicit. ACP cần: rearrange theo layer priority, thêm "Assumptions block" và "Senior engineer test".
FITS L1
ARCHITECTURE.md, DEVELOPMENT_PROTOCOL.md — already IS L1
ARCHITECTURE.md = perfect L1 artifact: module map, API contracts, dependency rules, convention table. Chỉ thiếu: ownership table và data classification. Không cần redesign, chỉ bổ sung.
FITS L2
policies.yml — nhưng govern agents, không govern Cursor
Đây là sự nhầm lẫn quan trọng nhất. ACP đã có excellent L2 cho AI agents nhưng hoàn toàn thiếu L2 cho Cursor-as-developer. Cần tạo CURSOR_RISK_POLICY.md như artifact L2 riêng.
FITS L3
BRANCH_PROTECTION.md + .cursorrules Rule 1
Có nhưng enforcement yếu. Rule 1 (1 PR/task) bị waive 2 lần (Sprint 1 + MC). Cần: diff size limit số cụ thể (≤300 LOC), file allowlist per task type, explicit sandbox rule (không modify master trực tiếp).
FITS L4
CI pipeline + test suite + DEVELOPMENT_PROTOCOL L5 gates
Strongest layer hiện tại. 156 tests, 8 SMK, fail_under=70, shipped parity = excellent L4. Thiếu nhỏ: security scan, contract tests cho TS bridge. Đây là gần production-grade nhất.
FITS L5
Claude HTML artifacts + Phase reports + audit docs = rich L5
ACP có L5 phong phú nhất so với L2. Decision log: tất cả Claude audit HTML. Change rationale: sprint reports. Failure patterns: audit final docs. Thiếu chỉ: structured extraction và formal feedback → .cursorrules loop.
Verdict: ACP fit với 6-layer tốt như thế nào?
ACP đã có L1, L4, L5 tốt nhất. L0 đã có 75% (cần refactor structure). L3 có nhưng không enforce. L2 hoàn toàn thiếu cho Cursor (chỉ có cho agents). Đây không phải rearchitecture từ đầu — đây là gap-fill và restructure khoảng 20–30% công sức, không phải 100%.
Nguyên tắc rearchitecture: Không discard artifacts đang hoạt động tốt. Restructure = reframe existing content + fill gaps. Karpathy principle "Simplicity First" áp dụng vào chính việc redesign này.
Phase 1 — Restructure L0 (.cursorrules)
Action
Rewrite .cursorrules thành 6 named sections khớp với layers
Hiện tại: 30 flat rules. Sau: 6 sections với priority ordering. L0 rules ở top = highest priority. L2 risk policy section explicit. L3 guardrails section với diff limit. Cursor sẽ parse theo thứ tự ưu tiên.
Thêm mới
Assumptions block trong mọi Cursor prompt template
"Before coding: state (1) what files will be touched, (2) what assumptions you're making, (3) what the verify command is. If any assumption unclear → ask, don't proceed." — Karpathy: "Don't assume. Don't hide confusion."
Effort
~2 giờ. Docs-only. 0 code risk.
Restructure .cursorrules + update Claude prompt template để include assumption block. Test: chạy 1 session Cursor với new .cursorrules, verify nó state assumptions trước code.
Phase 2 — Tạo CURSOR_RISK_POLICY.md (L2 critical gap)
LOW risk
Docs-only, test fixes, comment updates
Proceed without approval. Max 50 LOC. File allowlist: *.md, tests/, docs/. Forbidden: src/ changes. Verify: ruff + pytest unchanged count.
MEDIUM risk
New test files, CLI additions, config changes
State plan first. Max 200 LOC. Require verify gate pass before PR. Forbidden: touching core/policies.py, models.py without Claude review of changes.
HIGH risk
core/ changes, API schema changes, policy logic changes
Require Claude architecture review mid-task (not just post). Max 300 LOC. Mandatory: diff summary comment in PR. Forbidden: monolithic PR combining HIGH + LOW tasks.
CRITICAL
Invariant-touching changes, ABAC/PolicyEngine, identity auth
Human must explicitly approve before Cursor starts. Claude writes spec first. Cursor implements. Human reviews diff before merge. No exceptions.
Forbidden ops
Absolute list — Karpathy: "clearly say what not to do"
1. Direct push to master. 2. Remove or rename invariant rules. 3. Add OSS policy engine imports to core/. 4. Combine scope from different risk levels in 1 PR. 5. Commit without running verify gate.
Phase 3 — LESSONS_LEARNED.md (L5 → L0 feedback loop)
Pattern 1
Monolithic PR risk
Occurred: Sprint 1 PR #48 (MB-S1-1..5 + agent4), PR #63 (MC-1..11). Cause: no LOC limit in .cursorrules. Rule added: "HIGH risk tasks max 300 LOC, no combining with LOW tasks" → L3 guardrails.
Pattern 2
Scope creep in doc-only PRs
Occurred: agent4 config in PR #46 (P2-1 doc-only). Cause: no file allowlist for doc tasks. Rule added: "docs-only PR file allowlist: *.md, docs/, config/*.yml read-only" → L3 guardrails.
Pattern 3
Auto-close failure
Occurred: PR #63 Closes #52..#62 range not parsed. Cause: GitHub not parse range. Rule added: "List each issue separately in PR body: Closes #53, Closes #54..." → L5 process.
Pattern 4
Silent ABAC assumption
Occurred: load_policies() skip role_not_in silently (GAP-ABAC-2). Cause: no "state assumptions before coding" step. Rule added: "Before implementing any policy loader change, state which condition keys will be handled vs skipped" → L0 behavioral.
Tác động và rủi ro của rearchitecture

LOW risk .cursorrules restructure

Không có code risk. Cursor sessions hiện tại sẽ tốt hơn vì rules được prioritized. Downside: Cursor cần learn new structure — 1 session test trước khi dùng production.

MED risk New CURSOR_RISK_POLICY.md

Nếu risk classification quá strict → Cursor sẽ ask for approval quá nhiều. Cần calibrate: "biases toward caution over speed" (Karpathy) nhưng không block trivial tasks. Solution: LOW tier = no friction.

LOW risk LESSONS_LEARNED.md

Pure additive. Nếu maintained well, đây là tài sản governance quý nhất. Risk: không maintain → stale. Mitigation: thêm vào sprint-close checklist.

NONE Existing artifacts

Không discard bất kỳ artifact nào. ARCHITECTURE.md, CI pipeline, sprint reports đều kept. Rearchitecture là additive layer, không replacement.

MED Prompt template change

Mọi Claude prompt sang Cursor cần có "Assumptions block". Overhead ~5 phút per prompt. Benefit: eliminate silent assumption drift. Trade-off rõ ràng — worth it cho HIGH/CRITICAL tasks.

POSITIVE Overall system quality

L2 risk policy specifically addresses monolithic PR và scope creep — 2 vấn đề lớn nhất trong ACP history. Nếu implement đúng, sprint 2+ sẽ có cleaner commit history và fewer post-merge hygiene PRs.

Template .cursorrules mới — 6 sections, priority-ordered. Karpathy 4 principles làm L0 section đầu tiên. Hiện tại ACP .cursorrules có gì đúng → keep, thêm missing pieces theo layer.

# ai-control-plane .cursorrules — 6-layer governance # Priority: L0 > L1 > L2 > L3 > L4 > L5 # Conflict resolution: higher layer always wins # Last updated: post Milestone C (master @ 6dfffdf) ## L0 — Behavioral Constitution (Karpathy 4 Principles) # These apply to EVERY task, no exceptions THINK BEFORE CODING: - Before any code: state explicitly (a) files you will touch, (b) assumptions being made, (c) verify command - If uncertain about interpretation → ask, do NOT pick silently and run with it - If a simpler approach exists → say so before implementing the complex one - "Senior engineer test": would a senior say this is overcomplicated? If yes, simplify first. SIMPLICITY FIRST: - Minimum code that solves the problem. Nothing speculative. - No abstractions for single-use code; no "flexibility" not requested - If 100 lines could be 50 → propose the rewrite, don't add to 100 - No error handling for impossible scenarios SURGICAL CHANGES: - Touch ONLY what the task requires. Every changed line must trace to the request. - Do NOT "improve" adjacent code, comments, or formatting unless asked - If you notice dead code → mention it, do NOT delete it - Remove only imports/variables YOUR changes made unused (not pre-existing) GOAL-DRIVEN EXECUTION: - Every task must have explicit verify: [command] — state it before starting - Transform "fix the bug" → "write test reproducing it, then make it pass" - Multi-step tasks: state plan with verify per step before executing step 1 ## L1 — Project Context Package: ai_control_plane (src/ layout). CLI: agentctl (Typer). API: FastAPI api/server.py. Config: Pydantic v2 + PyYAML via ACP_CONFIG_DIR env var. Read ARCHITECTURE.md before any non-trivial task. It is the L1 authority. Module ownership (who must review changes): - core/policies.py → Claude architecture review required (CRITICAL) - core/models.py → Claude review required (HIGH) - api/server.py → Cursor-OK with invariant check (MEDIUM) - tests/, docs/ → Cursor-OK (LOW) - apex/ → Claude design spec required before implement (HIGH) Coding conventions: - Pydantic v2 (ConfigDict, not class Config) - structlog everywhere (not logging stdlib) - Async/await for all I/O. Type hints on all public signatures. - Docstring 1 line per public class/function. ## L2 — Risk Policy Task risk levels — MUST classify before starting: LOW: docs-only, test additions, comment fixes → proceed, max 50 LOC changed MEDIUM: new tests, CLI additions, config changes → state plan first, max 200 LOC HIGH: core/ changes, API schema changes, new endpoints → Claude review mid-task, max 300 LOC CRITICAL: PolicyEngine, ABAC, invariant-touching, identity auth → human explicit approve first Forbidden operations (absolute — no exceptions): 1. Direct push or merge to master 2. Remove, rename, or weaken any of the 8 invariants 3. Import OSS policy engines (CrewAI, LangChain, etc.) into core/ 4. Combine tasks of different risk levels in 1 PR 5. Run git commands that affect master without explicit human instruction 6. Commit without running the verify gate first PR size limits: LOW tasks: max 50 LOC. MEDIUM: max 200 LOC. HIGH: max 300 LOC. If task exceeds limit → split into multiple PRs, do not waive. ## L3 — Execution Guardrails Branch isolation: Every task = new branch from master. Name: {risk-level}/{issue-id}-{short-desc} Never reuse a branch for a different task scope (agent4-in-doc-PR pattern = FORBIDDEN). File allowlists per task type: docs-only task: allowed = [*.md, docs/**, config/*.yml (read-only)] forbidden = src/**, tests/** (any changes = scope creep, stop and ask) test-only task: allowed = [tests/**, tests/fixtures/**] forbidden = src/** (if test requires src change → it's a MEDIUM task, not LOW) core/ task: allowed = [src/ai_control_plane/core/**, tests/test_{module}.py] forbidden = api/**, mcp/**, cli/** (unless explicitly in task scope) Invariants (must check before every src/ PR): 1. core/policies.py: no OSS policy engine imports 2. core/models.py: all new types here only, in __all__ 3. mcp/git_server.py: facade only, no git logic 4. cli/: no direct core/policies.py import 5. apex/: all OSS tool calls FROM apex/, not the reverse 6. api/: sole cross-language bridge 7. core/quota.py: QuotaStore ABC, no hardcoded Redis 8. config/: ACP_CONFIG_DIR env var, no hardcoded paths ## L4 — Evaluation Mandatory verify gate (run in this order): ruff check src/ tests/ mypy src/ai_control_plane/ --strict pytest tests/ -v pytest tests/test_smoke.py -v -m smoke pytest tests/test_shipped_config_parity.py -v -m shipped_config Coverage requirements: fail_under = 70 (in pyproject.toml). core/policies.py >= 80%. cli/ >= 60%. Before declaring done: - All verify gate commands pass - No new mypy errors introduced (outside known apex stubs) - PR diff reviewed for orthogonal changes (Surgical Changes check) - GitHub issues linked in PR body (individual Closes #N, not ranges) ## L5 — Governance & Memory Decision log: docs/governance/*.html — Claude architect decisions archived here. Change rationale: docs/governance/PHASE*_REPORT.md per sprint. Failure patterns: docs/governance/LESSONS_LEARNED.md — update after each sprint. Lessons → rules loop: every failure pattern in LESSONS_LEARNED must reference which .cursorrules layer it maps to. Sprint close checklist: 1. All GitHub issues closed with individual Closes #N 2. Doc drift fixed (ARCHITECTURE.md, reports) 3. LESSONS_LEARNED.md updated with new patterns 4. .cursorrules reviewed for rule additions from lessons
